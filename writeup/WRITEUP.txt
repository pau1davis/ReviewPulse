WHAT I CUT AND WHY

1. ARCHITECTURE.md (N7 partial cut)
   Decided a thorough README with stack table, local setup, and environment
   variable docs delivered more value than a separate one-page architecture doc.
   The design decisions that matter (NullPool for Celery, pgvector for embeddings,
   Supabase Auth for session management) are commented inline where they live.
   With more time: write the one-pager explicitly answering "why not X" for each
   major choice (why pgvector over Pinecone, why Celery over FastAPI background
   tasks, why synthetic mode over a scraper).

2. Live deployment (N8)
   Didn't deploy to Render/Vercel before submission. Everything runs locally with
   one command. This is a real gap — a grader can't click through the UI without
   setup. With more time: Render for the FastAPI + Celery worker (both from the
   same Dockerfile with different CMD overrides), Vercel for the Vite frontend,
   Upstash Redis for the queue.

3. Observability dashboard (N12)
   Logs are structured and queryable but not wired to an external sink.
   With more time: pipe structlog JSON output to Axiom free tier; add a /metrics
   endpoint with Prometheus counters for job_started, job_completed, llm_cost_usd.

4. Real reviews dataset (alternative to synthetic mode)
   The spec explicitly names synthetic mode as the recommended path, so this was
   an intentional non-cut. But 25 reviews per book (vs the spec's 100–300) is
   a cut: trend lines are noisy at small N. Went with 25 to stay under free-tier
   Gemini quotas during development.


─────────────────────────────────────────────────────────────────────────────
THE ONE DECISION I'D REVERSE
─────────────────────────────────────────────────────────────────────────────

Using the google-generativeai SDK for embeddings instead of calling the REST API
directly from the start.

The SDK (v0.8.5) routes embed_content() through the v1beta endpoint, which doesn't
serve gemini-embedding-001 (the renamed text-embedding-004). This caused a silent
404 that only surfaced once reviews were in the DB with null embeddings, breaking
semantic search entirely. Switching to a direct httpx call to the v1 endpoint with
outputDimensionality=768 fixed it, but cost debugging time I'd have spent on
features.

If I started over: call the Gemini embedding REST API directly from day one.
The SDK is convenient for generation but adds an abstraction layer I can't control.


─────────────────────────────────────────────────────────────────────────────
WHAT'S INTERESTING ABOUT THE PROBLEM (BEYOND THE SPEC)
─────────────────────────────────────────────────────────────────────────────

The spec frames this as a data problem: ingest, analyze, surface metrics. But the
real problem is psychological. An independent author who gets a 1-star review at
9 PM doesn't need a dashboard — they need a triage protocol.

What's missing from the spec: a way for authors to *act* on their intelligence, not
just observe it. The "is_actionable" flag exists in the data model, but there's no
workflow for what the author does next. In practice, an author seeing 4 actionable
negative reviews wants to:
  - Mark one as "will address in next edition"
  - Draft a response to another
  - Dismiss one as bad faith
  - Flag one to their publisher

This is more like an inbox than a dashboard. The product-instinct addition I'd build
(and the thing I noticed that wasn't in the spec) is a Review Response Workspace: a
focused mode where the author sees only actionable reviews, can write a draft reply,
tag the review with a resolution status (acknowledged / drafting / responded /
dismissed), and track which criticism they've incorporated into their revision notes.

The LLM is already reading the review for sentiment and themes. The next obvious step
is asking it: "What would a gracious, professional author response look like?" That's
a one-sentence addition to the analysis pipeline that would close the loop from
"intelligence" to "action" — and it's the thing authors would actually pay for.


─────────────────────────────────────────────────────────────────────────────
HOW AI TOOLS SHAPED THE WORK
─────────────────────────────────────────────────────────────────────────────

Used GitHub Copilot (Claude Sonnet) throughout via VS Code.

Where it helped most:
  - Boilerplate: Pydantic schemas, SQLAlchemy models, FastAPI route stubs. These
    are mechanical and Copilot produces correct, idiomatic code for them quickly.
  - Test fixture structure: the pytest-asyncio session/function loop scope issue
    (asyncio_default_test_loop_scope) would have taken an hour to diagnose from
    docs alone; Copilot surfaced the right pytest.ini setting in one turn.
  - The HMAC webhook signing code: got it right on the first try, including the
    canonical JSON serialization (separators=(",", ":")).

Where I overrode it or had to intervene:
  - The embedding API bug: Copilot suggested re-trying with a different model name.
    I went a level deeper — read the actual HTTP response, traced it to the SDK
    routing through v1beta, and rewrote embed_text() from scratch with httpx.
  - The digest sentiment formula: Copilot's initial formula used an absolute delta
    (this_neg - this_pos > prev_neg - prev_pos + 1) which falsely flagged "declining"
    when both weeks were 100% positive but volume differed. Caught this by reading
    the logic, not from a test failure. Switched to a rate-based comparison.
  - Auth: ES256 JWKS validation required understanding Supabase's JWT key rotation
    behavior. Copilot offered HS256 by default; I had to specify ES256 and JWKS
    endpoint handling explicitly.

Net assessment: AI tools compress the mechanical 60% of the work to near-zero,
which frees time for the 40% that requires genuine debugging and product judgment.
The mistakes they make are subtle (wrong API version, off-by-one in a formula) and
only catchable if you read what they produce rather than accepting it on faith.
